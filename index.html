    Introduction to Clean Architecture Analyzer

    Clean Architecture Analyzer is a professional static code analysis toolset built on top of the Microsoft Roslyn compiler API. It is designed to strictly enforce clean architecture boundaries, DDD constraints, and SOLID principles in C# codebases.

    By running directly within the compiler pipeline (as part of MSBuild or IDE processes), the analyzer prevents architectural drift. In large development teams, developers frequently bypass design guidelines by introducing quick references to database contexts or external APIs directly inside core business objects. This analyzer blocks such violations at compile-time, acting as an automated architectural gatekeeper.

    💡 Why Enforce Architecture Statically?

    Without automated checks, software architectures inevitably degrade over time due to "architectural drift" — where developers introduce coupling that makes testing harder and locks the business logic to specific database engines or external APIs. Statically analyzing references ensures that rules are respected on every developer commit.

    Key Architecture Pillars Enforced

    • Dependency Direction: Enforces that dependencies point inwards. The core Domain has zero dependencies. The Application layer only depends on the Domain. Infrastructure and Presentation sit at the periphery and depend on the Application.
    • Domain Isolation: Domain models (Entities, Value Objects, Aggregates) must represent pure business models. They are forbidden from holding connection objects, performing direct database calls, or executing side-effects.
    • Encapsulation and Invariants: Objects must protect their state. Properties must have private setters, collections must be read-only, and state modifications should occur exclusively via explicit business methods.

    Installation & Setup Guide

    You can configure the analyzer to run locally within your projects or system-wide across your IDE. For teams, project-wide integration via NuGet is highly recommended to enforce the rules during command-line compilation and CI/CD pipelines.

    1. Visual Studio 2022 Integration

    Method A: NuGet Package (Project-Level Enforcer)

    Installing the analyzer via NuGet attaches it directly to the build pipeline. Any architectural violation will generate a compile-time warning (or error if configured) in any environment, including local terminals and cloud builders.

    Run the following command in the Package Manager Console or your system shell: 

    dotnet add package CleanArchitectureAnalyzer

    Once installed, you will see the analyzer listed under the Analyzers dependency tree inside Visual Studio Solution Explorer for each configured project.

    Method B: VSIX Extension (Design-Time Enforcer)

    If you want design-time warnings across all C# projects without modifying the .csproj file, you can install the IDE extension:

    1. Launch Visual Studio 2022 and navigate to Extensions > Manage Extensions.
    2. Select the Online tab and search for Clean Architecture Analyzer.
    3. Click Download. Visual Studio will queue the installation.
    4. Close all instances of Visual Studio. The VSIX installer will start automatically to complete the setup.

    2. VS Code (Visual Studio Code) Integration

    VS Code uses Roslyn analyzers dynamically through OmniSharp or the C# Dev Kit. To activate the warnings:

    1. Install the C# Dev Kit extension.
    2. Ensure you add the NuGet package in your project files (Method A above).
    3. Verify that Roslyn analyzers are enabled inside your settings.json: 
      "omnisharp.enableRoslynAnalyzers": true,
"csharp.showRoslynAnalyzers": true
    4. Restart the C# Language Server. The diagnostics will begin appearing in your "Problems" panel.

    Configuration Reference

    Because Clean Architecture is implemented across multiple assemblies, the analyzer must identify which C# project belongs to which architectural layer. It does this using the following settings:

    1. Setting Up .editorconfig (Recommended)

    The most reliable method is to create an .editorconfig file at the root of your solution. Add global options that explicitly define the assembly names for each layer. The analyzer reads these settings during compilation: 

    is_global = true

# Define assembly name rules (matches exact name or prefix matches)
cleanarchitecture_domain_assembly = MyCompany.ProjectName.Domain
cleanarchitecture_application_assembly = MyCompany.ProjectName.Application
cleanarchitecture_infrastructure_assembly = MyCompany.ProjectName.Infrastructure

    By using this configuration, even if you split your layers across multiple projects (e.g., MyCompany.ProjectName.Domain.Core and MyCompany.ProjectName.Domain.Events), the suffix matching will group them under the correct layer constraints.

    2. Naming Conventions (Fallback)

    If no .editorconfig configuration is detected at compilation time, the analyzer falls back to convention-based assembly matching. It inspects the assembly name and groups projects based on their suffix:

    • Domain: Assembly names ending with .Domain or exactly named Domain.
    • Application: Assembly names ending with .Application or exactly named Application.
    • Infrastructure: Assembly names ending with .Infrastructure or exactly named Infrastructure.
    ⚠️ Warning on Naming

    If your project names do not follow the standard suffixes (e.g., MyCompany.BusinessLogic instead of MyCompany.Domain) and you do not configure an .editorconfig, the analyzer will not execute layer checks on those assemblies.

    Domain Conventions & Suffix Matching

    Inside the Domain layer, DDD rules apply differently depending on whether a class is an Entity, a Value Object, an Aggregate Root, a Domain Service, or a Domain Event. The analyzer evaluates the C# type tree and classifies symbols based on standard naming suffixes and namespace structures:

    Matching Suffix Rules

    The type classification is resolved using the class name structure:

    • Entity: Detected if the class name ends with Entity (e.g., CustomerEntity, OrderEntity).
    • Value Object: Detected if the class name ends with Object (e.g., MoneyObject, AddressObject).
    • Aggregate Root: Detected if the class name ends with Root (e.g., OrderRoot, InventoryRoot).
    • Domain Service: Detected if the class name ends with Service (e.g., DiscountService, TaxCalculationService).
    • Domain Event: Detected if the class name ends with Event (e.g., OrderShippedEvent) or if the class is located inside a namespace ending with .Events or .Event.
    💡 DDD Architectural Tip

    Always name your Domain types strictly according to their role. An incorrectly named Value Object (e.g., named AddressInfo instead of AddressObject) will bypass Value Object-specific checks like immutability audits.

    Domain Layer Rules Reference (30 Rules)

    The Domain layer represents the core business capabilities. It must remain pure, highly testable, and completely decoupled from technical frameworks.

    CAA0001 Warning
    Domain elements depends on External Classes
    Rationale: Domain elements must have no dependencies on outer frameworks or libraries. Referencing external classes directly makes domain logic rigid and dependent on outer implementation details.

    Violation: Triggered when a field, property, or return type inside a Domain class references a type outside of the Domain project or System libraries.
    ❌ Bad
    public class OrderEntity
{
    // Bad: Direct dependency on EF Core Context
    private SQLServerContext _context;
}
    ✔️ Good
    public class OrderEntity
{
    // Good: Pure encapsulation
    private readonly Guid _id;
}
    CAA0002Warning
    Domain elements must not contain dependency injection in constructor
    Rationale: Business entities should be constructed using their initial state (e.g., ID, name, items). Injecting services inside entities hides side-effects and prevents unit testing in isolation.

    Violation: Triggered when any constructor inside a Domain class accepts a parameter whose type is classified as external or service-oriented.
    CAA0003 Warning
    Domain elements should not expose collections as List<T>
    Rationale: Exposing List<T> allows external layers to add or clear items directly, bypassing the domain entity's invariant logic.

    Violation: Exposing a public property of type List<T>, ICollection<T>, or IEnumerable<T> that has public setters.
    ❌ Bad
    public class OrderEntity
{
    public List<OrderItem> Items { get; }
}
    ✔️ Good
    public class OrderEntity
{
    private readonly List<OrderItem> _items;
    public IReadOnlyCollection<OrderItem> Items => _items.AsReadOnly();
}
    CAA0004Warning
    Domain elements should not depend on external interfaces
    Rationale: Implementing external contracts couples the Domain to outer structures, making changes difficult.

    Violation: Implementing interfaces originating from Infrastructure or Application layers.
    CAA0005Warning
    The entity should not expose public builders without parameters
    Rationale: Forcing constructors to accept parameters guarantees that the Entity is initialized in a valid state. Parameterless constructors should be non-public.

    Violation: Declaring a public parameterless constructor on an entity.
    CAA0006Warning
    Entities should not expose public constructors accepting an ID
    Rationale: Entity identities should be generated internally (or via DDD factories) to prevent breaking uniqueness invariants.

    Violation: Public constructor accepting a parameter of type Guid or named with the suffix Id.
    CAA0007Warning
    Domain event handlers should not be placed in the Domain layer
    Rationale: Domain event handling involves cross-cutting concerns (DB updates, notifications, external APIs). They must reside in Application/Infrastructure.

    Violation: Declaring a class implementing event handler interfaces (e.g., INotificationHandler) in Domain.
    CAA0101Warning
    Possible Anemic Entity
    Rationale: OOP requires grouping data and behaviors. Anemic entities are simply database tables in memory, which leads to procedural code.

    Violation: Defining an entity with 4 or more properties but 0 business methods.
    CAA0102Warning
    Entity with Public Setters
    Rationale: Public setters allow external code to bypass state validation logic, leading to invalid states.

    Violation: Declaring properties on an Entity with public set accessors.
    CAA0103Warning
    Entity should have an Id
    Rationale: By definition, entities have an identity.

    Violation: An Entity class missing an Id property.
    CAA0201 Warning
    Value object should be immutable
    Rationale: Value Objects must represent static values. State modifications are achieved by instantiating new objects, ensuring thread safety and preventing bugs.

    Violation: Defining mutable properties on a Value Object class.
    ❌ Bad
    public class AddressObject
{
    public string City { get; set; }
}
    ✔️ Good
    public class AddressObject
{
    public string City { get; }
}
    CAA0202Warning
    Value object must not have an ID
    Rationale: Value Objects are identified solely by their attributes, not an ID. Adding an ID turns the object into an Entity.
    CAA0203Warning
    Value object should implement value equality
    Rationale: Direct comparison must evaluate attribute equality, not memory references.

    Violation: Missing Equals and GetHashCode overrides.
    CAA0204Warning
    Value object should not contain collections
    Rationale: Value objects should be simple structures and should not contain lists of mutable objects.
    CAA0205Warning
    Value objects must implement IEquatable<T>
    Rationale: Reduces performance overhead during comparison by avoiding boxing.
    CAA0206Warning
    Value object collections must be ReadOnly
    Rationale: Exposes read-only collections to enforce immutability constraints.
    CAA0207Warning
    Value object must not reference entities or aggregate roots
    Rationale: Prevents holding references to objects with mutable lifecycles.
    CAA0301Warning
    Aggregate root cannot have properties of a type other than aggregate root
    Rationale: Aggregates represent transactional boundaries. Direct referencing violates aggregate autonomy.
    CAA0302Warning
    Domain events should only be raised by Aggregate Roots
    Rationale: To maintain consistency, child entities should delegate event registrations to their parent Aggregate Root.
    CAA0303Warning
    Aggregate roots should use Factory methods instead of public constructors
    Rationale: Restricts creation to explicit factories, ensuring invariants are satisfied during instantiation.
    CAA0401Warning
    Domain services should be stateless
    Rationale: Domain services represent operations and must be stateless to guarantee concurrent safety.
    CAA0402Warning
    Domain services should not expose mutable properties
    Rationale: Prevents storing mutable state within operational domain service instances.
    CAA0403Warning
    Domain services should implement at least one interface
    Rationale: Facilitates mockability in unit testing and supports dependency injection registration.
    CAA0404Warning
    Domain services should not be static classes
    Rationale: Enforces object-oriented design and decouples logic from static references.
    CAA0405 Warning
    Domain service methods must not accept or return DTOs or ViewModels
    Rationale: Exposing DTO contracts inside Domain services leaks external API or UI layouts into the pure Domain.
    ❌ Bad
    public UserDto GetUser(UserRequest req);
    ✔️ Good
    public UserEntity GetUser(Guid userId);
    CAA0406Warning
    Domain services must not depend on concrete repository implementations
    Rationale: Concrete repositories link the service directly to database frameworks. Inject abstractions instead.
    CAA0501Warning
    Domain events must be immutable
    Rationale: Domain Events represent past events. Past facts cannot be modified.
    CAA0502Warning
    Domain events must contain a timestamp property
    Rationale: Knowing when an event occurred is critical for debugging, logs, and event sourcing.
    CAA0503Warning
    Domain events must not reference mutable entities or aggregate roots directly
    Rationale: Events are serializable. Referencing full entity objects causes cycle dependency errors.
    CAA0504Warning
    Domain events should not contain business logic methods
    Rationale: Events are data records. Logic operations belong in Services or Use Cases.
    CAA0505Warning
    Domain events should end with Event or be named in the past tense
    Rationale: Clearly identifies that the class represents a historical fact.

    Application Layer Rules Reference (20 Rules)

    The Application layer defines the use cases. It acts as an orchestrator, dispatching requests and coordinating business models.

    CAA1001 Warning
    Application layer must not reference Infrastructure
    Rationale: Referencing concrete persistence classes couples Application to specific DB servers.

    Violation: Declaring a field or method parameter of an Infrastructure type inside Application classes.
    ❌ Bad
    using MyProject.Infrastructure;
public class Service { SQLRepository _repo; }
    ✔️ Good
    using MyProject.Application.Interfaces;
public class Service { IOrderRepository _repo; }
    CAA1002Warning
    Application layer must not reference Presentation
    Rationale: Application must remain agnostic of delivery mechanisms (e.g. Web API, MVC Controllers).
    CAA1003 Warning
    Use case request must be immutable
    Rationale: Request payloads must not mutate mid-flight. Read-only properties guarantee thread-safety.
    ❌ Bad
    public class CreateCommand { public string Name { get; set; } }
    ✔️ Good
    public class CreateCommand { public string Name { get; } }
    CAA1004Warning
    Use case handler should not expose public properties or fields
    Rationale: Handlers are executor objects. Exposing public fields invites external state tampering.
    CAA1005Warning
    Use case output must not expose Domain Entities or Aggregate Roots
    Rationale: Returning entities directly couples outer clients to the domain model structure. Map entity outputs to DTOs.
    CAA1006Warning
    Use case input must not accept Domain Entities or Aggregate Roots
    Rationale: Direct parameter mapping causes API clients to send entity models, exposing Domain internals.
    CAA1007Warning
    Application services must be stateless
    Rationale: Application service models coordinate processes. Holding states causes concurrent errors.
    CAA1008Warning
    Validators must reside in the Application layer
    Rationale: Application validators validate request models before use case execution.
    CAA1009Warning
    Application exceptions should be sealed
    Rationale: Prevents inheritance and supports clean exception management.
    CAA1010Warning
    CQRS Commands should not contain business logic or behavior
    Rationale: Commands carry action data. Processing belongs strictly in Handlers.
    CAA1011Warning
    CQRS Queries should not contain business logic or behavior
    Rationale: Queries retrieve data. Business checks belong in Domain.
    CAA1012Warning
    Application layer should not access the file system directly
    Rationale: Disk access is an Infrastructure concern. Abstraction allows test mockups.
    CAA1013 Warning
    Application layer should not use DateTime.Now or DateTime.UtcNow directly
    Rationale: Calling system clock directly prevents testing relative time calculations deterministically.
    ❌ Bad
    var date = DateTime.UtcNow;
    ✔️ Good
    var date = _dateTimeProvider.UtcNow;
    CAA1014Warning
    Command name should start with a verb
    Rationale: Commands represent direct requests. Start names with verbs (e.g. CreateOrder).
    CAA1015Warning
    Query name should end with Query or start with Get/Find/List/Search
    Rationale: Enforces uniform naming to distinguish read queries from write actions.
    CAA1016Warning
    Do not use Task.Delay or Thread.Sleep inside Use Cases
    Rationale: Hardcoded delays block thread execution, causing throughput issues.
    CAA1017Warning
    Use case handlers must not contain empty catch blocks
    Rationale: Empty catch statements swallow diagnostics, hiding programming bugs.
    CAA1018Warning
    DTO classes should not contain behavior/business logic methods
    Rationale: DTOs must carry data, not operations.
    CAA1019Warning
    Application layer must not reference third-party SDK clients directly
    Rationale: Direct referencing creates tight coupling to external vendors (e.g. Stripe, SendGrid).
    CAA1020Warning
    Use case handlers should be sealed
    Rationale: Use Case Handlers are leaf nodes representing execution paths and should not be extended.

    Infrastructure Layer Rules Reference (20 Rules)

    The Infrastructure layer contains technology implementation details, database contexts, file handlers, and mail servers.

    CAA2002 Warning
    Infrastructure implementation classes should be sealed
    Rationale: Restricts class extension, protecting technical configurations from unexpected changes.
    ❌ Bad
    public class SQLServerService : IService
{
}
    ✔️ Good
    public sealed class SQLServerService : IService
{
}
    CAA2003Warning
    DbContext or DbConnection must not be exposed
    Rationale: Keeps database details private to Infrastructure. Exposing DbContext allows other layers to query DB tables directly.
    CAA2012 Warning
    Use ILogger instead of System.Console
    Rationale: Console outputs cannot be structured, redirected, or filtered easily in production environments.
    ❌ Bad
    Console.WriteLine("Error occurred");
    ✔️ Good
    _logger.LogError("Error occurred");
    CAA2018 Warning
    Repositories should not save or commit transactions internally
    Rationale: Calling SaveChanges inside repositories commits database transactions prematurely. Delegate it to Unit of Work.
    ❌ Bad
    _context.SaveChanges();
    ✔️ Good
    // Delegated to Unit of Work
    CAA2001Warning
    Infrastructure classes must not contain core business logic
    Rationale: Limits responsibilities. Infrastructure classes should only contain adapter logic.
    CAA2004Warning
    Repositories must be internal or sealed
    Rationale: Restricts access to concrete DB query operations, ensuring code uses interfaces.
    CAA2005Warning
    Database entities/persistence models must not be exposed outside Infrastructure
    Rationale: Database models are mapped to database structures. Exposing them couples outer projects to database schemas.
    CAA2006Warning
    Avoid raw/interpolated SQL strings in Infrastructure without parametrization
    Rationale: Using raw non-parameterized SQL statements creates vulnerabilities to SQL Injection.
    CAA2007Warning
    Repositories should not perform business validation
    Rationale: Ensures repository operations focus strictly on data persistence.
    CAA2008Warning
    HTTP Clients and External API calls must only reside in Infrastructure
    Rationale: External API network requests are integration details and belong in Infrastructure.
    CAA2009Warning
    Infrastructure services must be thread-safe
    Rationale: Singleton or scoped service instances must remain stateless to prevent threading anomalies.
    CAA2010Warning
    Connection strings or API secrets must not be hardcoded
    Rationale: Hardcoding connection credentials causes severe security vulnerabilities. Use configuration providers.
    CAA2011Warning
    DbContext classes should inject DbContextOptions
    Rationale: Enables configuring connection strings dynamically during DI registration.
    CAA2013Warning
    Classes with Repository suffix must implement a repository interface
    Rationale: Enforces Dependency Inversion, decoupling code from concrete repository classes.
    CAA2014Warning
    Infrastructure classes should have recognized suffixes
    Rationale: Suffix identifiers (e.g. Repository, Adapter) ensure clean architectural roles.
    CAA2015Warning
    Migrations must not be in Domain or Application layers
    Rationale: Database schema versions are persistence details and belong in Infrastructure.
    CAA2016Warning
    Infrastructure services must not expose parameterless constructors if dependencies exist
    Rationale: Forces client code to use dependency injection, preventing uninitialized states.
    CAA2017Warning
    EF Core queries in read repositories should use AsNoTracking
    Rationale: Disabling change tracking improves execution performance on read queries.
    CAA2019Warning
    Infrastructure classes must not implement Domain Event or Entity interfaces directly
    Rationale: Infrastructure elements represent technology details, not business domain entities.
    CAA2020Warning
    Database and I/O operations should be asynchronous
    Rationale: Asynchronous execution prevents blocking thread pools, maximizing backend throughput.
